# The Atelier — 작업 지침

누리님의 작업 스타일과 자동화 규칙이 담겨 있습니다.

---

## 🚀 세션 시작 시 자동 실행

세션이 시작되면 **묻지 말고 자동으로** 다음을 실행하세요:

1. `git pull` 실행
2. `WORKLOG.md` 파일의 맨 위에서부터 최근 3개 세션 항목을 읽기
3. 누리님께 한국어로 짧게 보고:
   - 가장 최근 세션이 언제, 어느 기기에서였는지
   - 마지막에 "다음 할 일"로 남긴 항목
   - "막힌 점"이 있다면 그것도 함께
4. "오늘은 어디서부터 이어갈까요?" 같은 질문으로 마무리

## 🏁 세션 종료 시 자동 실행

누리님이 "오늘 작업 마무리", "푸시해줘", "세션 끝", "정리하자" 같은 말을 하면:

1. 오늘 작업 내용을 `WORKLOG.md` **맨 위에 추가** (prepend)
   - 템플릿은 WORKLOG.md 상단의 형식을 그대로 따르기
   - 작업 기기는 누리님께 짧게 물어보기 (맥북 / 아이맥 / 윈도우)
2. 변경된 코드 파일들과 WORKLOG.md를 `git add`
3. 의미 있는 커밋 메시지로 `git commit`
   - 예: `feat: AI Coach stage 5 자연어 챗 초안 + WORKLOG`
4. `git push`
5. 완료 보고

## 📐 작업 톤 & 규칙

### 코드 수정은 BEFORE/AFTER 형식

```
BEFORE:
  (수정 전 코드)

AFTER:
  (수정 후 코드)

이유: (왜 바꿨는지 한 줄)
```

### 평이한 용어 사용

- 누리님은 정규 코딩/이과 교육 배경이 없음
- "비동기 처리", "메모이제이션" 같은 전문 용어는 풀어서 설명
- 비유나 일상 예시로 먼저 감 잡게 한 다음 정확한 용어 알려주기

### Diagnosis-First 패턴

수정 요청을 받으면 **바로 고치지 말고**:

1. **읽기 전용 진단 먼저** — 관련 파일들 읽고 현재 상태 파악
2. **누리님께 진단 보고** — "지금 이런 상태이고, 이 부분이 원인 같아요"
3. **수정안 제시** — "이렇게 고치려는데 괜찮을까요?"
4. **승인 후 수정** — 그제서야 실제로 파일 수정

급한 단순 오타 수정 등은 예외. 하지만 로직이 얽힌 곳은 반드시 진단 먼저.

### 하지 말 것

- "물론이죠!", "좋은 질문이에요!" 같은 sycophantic 표현
- 과도한 이모지 (강조 1개 정도는 OK)
- 길고 장황한 설명
- 모달/팝업 디자인 제안 (누리님은 인라인 편집 + 하드코딩 기본값 선호)

- 단계별 진행
